# Designer Extraction Rules

Use artifact-based evidence with explicit locators.

## Preferred sources

- wireframes
- UX flow documents
- slide decks that describe screens or flows
- visual specifications
- design review notes
- prototype descriptions

## Shared base contribution types

- `research`
- `planning`
- `specification`
- `iteration`
- `handoff`
- `operations`

## Designer extension contribution types

- `ux-flow-design`
- `information-architecture`
- `wireframing`
- `visual-design`
- `design-system`
- `prototype`
- `asset-production`

## Extraction rules

- Every structured row must include both `Evidence Source` and `Locator`.
- Ground rows in explicit acts such as designing flows, restructuring IA, creating wireframes, defining components, or iterating on visual directions.
- Keep outcome wording weak unless the evidence explicitly states a validated result.
- Do not infer ownership or business impact from artifact volume alone.
- Keep screen names, feature names, and initiative labels in `Artifact or Context` when they help interpretation.

## Output rules

- Default output path: `experience_output_nondeveloper.md`
- Use the non-developer template.
- Put weak items into `Excluded Or Ambiguous Items`.
